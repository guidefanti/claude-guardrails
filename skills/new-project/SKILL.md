# New Project

You are a project configurator. Your task is to analyze the current project and
generate (or improve) a CLAUDE.md with integrated guardrails.

## Step 1: Detect current state

Check if a CLAUDE.md already exists at the project root:

```bash
ls -la CLAUDE.md 2>/dev/null && echo "EXISTS" || echo "NOT_FOUND"
```

### If CLAUDE.md does NOT exist:
- Proceed to Step 2 (analysis) and Step 3 (full generation)

### If CLAUDE.md EXISTS:
- Read current contents with `cat CLAUDE.md`
- Evaluate maturity:
  - **Generic** (generated by `/init`, < 30 lines, no known failures): propose
    full replacement with guardrailed template, ask for confirmation
  - **Mature** (has known failures, SSOT map, specific rules): propose
    ONLY missing additions (self-improvement, skill routing), NEVER remove
    or alter existing content. Show suggested additions and ask for confirmation
- **NEVER overwrite a mature CLAUDE.md without explicit approval**

## Step 2: Analyze the project

Collect real project context:

```bash
# Folder structure (2 levels)
find . -maxdepth 2 -type f \( -name "*.json" -o -name "*.toml" -o -name "*.yaml" -o -name "*.yml" -o -name "*.py" -o -name "*.ts" -o -name "*.tsx" -o -name "*.rs" -o -name "*.md" \) | head -40

# Package managers and configs
ls package.json Cargo.toml requirements.txt pyproject.toml composer.json Gemfile go.mod 2>/dev/null

# Recent git history
git log --oneline -10 2>/dev/null

# Available skills in the project
ls .claude/skills/*/SKILL.md 2>/dev/null

# Existing docs
ls docs/*.md README.md TODO.md CHANGELOG.md 2>/dev/null
```

### Detect project type (if $ARGUMENTS not provided):

| Indicator | Type |
|---|---|
| package.json, tsconfig, Cargo.toml, requirements.txt | dev |
| Folders like campaigns/, ads/, creatives/, funnels/ | marketing |
| n8n workflow .json files, automation/ folders | automation |
| Predominantly .md files, docs/, research/ | research |
| Folders like content/, posts/, scripts/ (screenplay) | content |

If $ARGUMENTS was provided, use the given type.

## Step 3: Generate CLAUDE.md

Use the template below, adapting to detected type and context.

### Base template:

```markdown
# [Project Name]

## Required skills

Load BEFORE starting any task:

- **Coding/development tasks:** [list relevant dev skills found, or suggest dev-guardrails]
- **Analysis/strategy/research tasks:** universal-thinking
- [Other relevant skills found in the project]

## Project context

[2-5 lines describing: what it is, who it's for, main stack/context.
 Fill based on Step 2 analysis.]

## Constraints and conventions

[List detected conventions: code language, framework, naming patterns,
 folder structure. If none detected, leave as placeholder.]

## Self-improvement

When the user corrects an error or points out a failure, log the pattern
below under "Known failures" with: (1) what happened, (2) what should
have been done, (3) the lesson learned. Do not ask, just do it.

## Known failures (DO NOT REPEAT)

[Empty — will be populated automatically as the project evolves.]
```

### Type-specific additions:

**dev:** Add "Environment" section (venv, build commands, test commands) and
"Subsystem map" if the project has multiple modules.

**marketing:** Add "Metrics and KPIs" and "Active channels" sections.

**automation:** Add "Active workflows" and "Integrations" sections.

**research:** Add "Primary sources" and "Methodology" sections.

**content:** Add "Tone and voice" and "Publishing channels" sections.

## Step 4: Present and confirm

- Show the generated CLAUDE.md to the user
- Briefly explain what each section does
- Ask for confirmation before saving
- Only after confirmation, write the file to the project root

## Hard rules

- NEVER generate a CLAUDE.md longer than 80 lines. Less is better.
- NEVER fabricate information about the project. If not detected, use placeholder.
- NEVER overwrite an existing CLAUDE.md without explicit approval.
- NEVER include instructions that a linter or hook can enforce.
- Fill what you can detect, leave the rest as clear placeholders.
